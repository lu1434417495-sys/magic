A2 战斗指令区重建详细计划

日期：2026-05-16
来源：docs/discussions/battle_map_ui_shortcomings.md 中 A2 条目。
状态：实现前的拆解，不包含代码修改。

1. 目标与范围

恢复 BottomPanel / TopBar 上被删但未重建的 UI 入口，把已经存在的命令通道暴露给玩家。不新增命令；这是纯渲染层补齐。

设计前提：战斗是策略游戏，节奏故意拉慢——每次选技能都是一次审慎决策，不追求"手不离键盘"的速度感。基于此前提：
  • 技能槽取消 1-9 键盘绑定，仅支持鼠标点击。槽数可无限扩展，不再受键位上限约束。
  • 指令按钮（结算 / 重置移动 / 切变体 / 取消）保留鼠标为主、键盘为辅的双通道——这些是高频战术节奏控制，键盘加速合理。
  • 视口平移（WASD）保留键盘——属于摄像机操作，与策略决策无关。

顺带扫掉的关联项：

  • C4 ModeValueLabel 改 ModeButton（点击切换 control_mode）
  • A1 残留：selected_skill_confirm_ready / selected_skill_auto_cast_ready 挂到结算按钮高亮态
  • E2 面板内精简战报回显（LogLabel）
  • C2 全局键位说明位（按钮自带 hotkey 文本）

3. snapshot 已就绪的字段（adapter 已产出）

battle_hud_adapter._build_hud_snapshot 返回里已有：

  • mode_text（manual / auto / ...）→ ModeButton 显示文本
  • selected_skill_confirm_ready → 结算按钮高亮态
  • selected_skill_auto_cast_ready → "自动施法可用"提示
  • selected_skill_target_count / target_min_count / target_max_count → 命令摘要 "2/3 目标"
  • skill_title / skill_subtitle → 已渲染到 SkillPanel，可不再重复

需要新增的 adapter 字段

  • recent_battle_log_lines: Array[String] —— 给面板内 LogLabel 用，取 RuntimeLogDock 同源 log，前 N 条精简版。
  • hint_text: String —— 当前可用操作的一行提示（如"按 Enter 结算 / Esc 取消选择"），来自当前 selection state 状态机。
  • command_dock: Dictionary —— 单独打包按钮启用态，避免面板自己拼：
command_dock = {
    "resolve_enabled": bool,         # confirm_ready && not modal
    "reset_movement_enabled": bool,  # 本回合有移动累计
    "clear_skill_enabled": bool,     # selected_skill_id != ""
    "prev_variant_enabled": bool,    # current_variant > 0
    "next_variant_enabled": bool,    # current_variant < max
    "mode_cycle_enabled": bool,      # ai / manual 切换允许
}

4. 节点 / 信号设计

被删除的历史节点名（来自 A2 原始列表）：ResolveBattleButton、ResetMovementButton、PrevVariantButton、NextVariantButton、ClearSkillButton、CommandDock、CommandSummaryLabel、HintLabel、LogLabel、UnitDetailLabel。

建议恢复 8 个节点，组织成 3 个区块：

区块 A — TopBar 右端（C4 顺带做）

TopBar/RightCell
├─ ModeButton (Button)            # 取代 ModeValueLabel；点击 cycle control_mode
└─ ResolveBattleButton (Button)   # 状态：confirm_ready → 主色亮 / 否则灰

区块 B — BottomPanel 中央偏右（指令带）

BottomPanel/CommandDock (HBoxContainer)
├─ ClearSkillButton (Button)      # 显示 "Esc 取消选择"，selected_skill_id 非空时亮
├─ PrevVariantButton (Button)     # ← 显示 "Q"
├─ VariantNameLabel               # 当前 variant 名（从 skill_title 拆出来）
├─ NextVariantButton (Button)     # → 显示 "E"
├─ ResetMovementButton (Button)   # "重置本回合移动"
└─ CommandSummaryLabel            # "已选 2/3 目标 · 50 STA"

区块 C — BottomPanel 右侧（信息位）

BottomPanel/InfoColumn (VBoxContainer)
├─ HintLabel    # 一行可执行操作提示，来自 snapshot.hint_text
└─ LogLabel     # 最近 3-4 条精简战报，固定高度

信号约定

每个按钮发面板级 signal，由 world_map_system 统一接 facade：

signal battle_resolve_pressed
signal battle_reset_movement_pressed
signal battle_cycle_variant_pressed(step: int)
signal battle_clear_skill_pressed
signal battle_mode_toggle_pressed

参考 world_map_system.gd:136 已有的 battle_skill_slot_selected.connect(...) 模式，不直接把 facade 注入面板——保持 panel 纯渲染。

6. 复杂度估算

  • 代码：约 +600 / -50 行
  • 场景：1 个 .tscn 加 8 个节点 + 1 个 HBox 容器
  • 测试：snapshot regression 字段加 6-8 个新 expectation；UI fixture 加按钮点击路径
风险点
  • command_dock 启用态的 source of truth 必须只在 adapter，否则面板会出现"按钮亮但 facade 拒绝"的不一致。
  • LogLabel 与 RuntimeLogDock 同源——需要决定是 dock 转发还是各自从 facade 拉。建议各自拉，避免互相依赖。
  • reset_movement 如果发现需要新增 facade 命令，把这一项拆成单独前置 PR。

7. 建议拆 PR

A2 工作量偏大，建议 3 个 PR：

  1. A2.1 — adapter 新增 3 个 snapshot 字段 + game_text_snapshot_renderer 同步 + regression 加 expectation。纯数据层，不动 UI。
  2. A2.2 — 场景节点 + 面板渲染逻辑 + signal + world_map_system 接线 + UI regression。
  3. A2.3 — C4 ModeButton 切换 control_mode + A1 残留 confirm_ready 灯。

A2.1 落地后，A2.2 / A2.3 可以并行评审；A2.1 是独立可测的数据补齐，风险最低。

8. 实现前需要先决议

  1. reset_movement 是否需要新 facade 命令？ 现状 battle_session_facade 没有暴露重置本回合移动累计的 API，必须先排查 battle_selection 是否已有内部方法，没有则前置加一个 command_battle_reset_movement。
  2. hint_text 状态机定义 至少覆盖 5 个状态：无技能选中 / 技能已选未达最小目标数 / 已达最小目标数可施法 / 模态阻断（战斗结算中）/ 自动模式。adapter 需要明确这块的状态来源（battle_selection 还是 battle_state.modal_state）。
  3. LogLabel 行数与时机 默认 3 行 + 战斗事件触发时刷新。需要决定是否在面板上做"点开看完整 log"的链接（推荐不做，留给 RuntimeLogDock）。
