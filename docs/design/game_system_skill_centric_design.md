## Summary

- 当前项目应明确把“技能系统”定义为**玩法语义核心**。
- `GameSession`、`GameRuntimeFacade`、`BattleRuntimeModule`、`CharacterManagementModule` 依然都是核心系统，但它们分别承担的是状态、编排、执行、成长写入职责，不替代技能系统的玩法本体地位。
- 从整游戏角度看，技能不是“战斗里的一个子功能”，而是角色成长、构筑表达、战斗执行、敌人决策、奖励投放、内容扩展的共同语言。
- 今后新增任何子系统，都应该先回答 3 个问题：
  - 它改变了什么技能状态。
  - 它消费了什么技能状态。
  - 它如何把结果回流到技能循环。

## 1. 设计命题

本项目的整游戏系统，不应以“职业”“身份”“剧情标签”或“数值成长”作为最高玩法主语，而应以“技能”作为最高玩法主语。

更准确地说：

- 技能定义单位**会什么**。
- 技能成长定义单位**变强了什么**。
- 技能组合定义单位**构筑成什么**。
- 技能投影到战斗里，决定单位**此刻能做什么**。
- 技能作为奖励与内容投放单位，决定玩家**接下来追求什么**。

所以本项目的系统总纲应是：

> 技能系统是玩法核心；其它核心系统围绕技能系统完成持久化、编排、执行、投放、反馈与验证。

## 2. 核心系统定位

### 2.1 顶层：玩家成长

本项目的玩法本体是**玩家成长**——玩家对队伍的持续投入，沉淀为正式可持久化的能力变化，反过来改变玩家在世界里能做什么。成长不是单一维度，而是由若干条正式支柱组成。

### 2.2 成长的五条支柱

| 支柱 | 语义 | 真源 Owner | 与其它支柱的主要交互 |
| --- | --- | --- | --- |
| 技能构筑 | 单位会做什么、构筑如何表达 | `ProgressionContentRegistry` + `PartyMemberState.skill_records` | 构筑语义核心；其它四条支柱都应向它提供门槛 / 修饰或从它消费信息 |
| 属性成长 | 单位强度基线与派生数值 | `UnitBaseAttributes` + `AttributeService` | 为技能执行提供数值底座，接受职业 / 装备修正 |
| 职业身份 | 构筑的组织、承接、放大方式 | `PartyMemberState.profession_records` | 用技能 / 属性做资格判定；反过来放大某组技能、提供装备门槛 |
| 装备构筑 | 外挂能力、数值放大、限制 | `EquipmentState` + `PartyEquipmentService` | 向属性支柱写入修正；向技能支柱提供门槛 / 授予 / 标签修饰 |
| 资源与世界进度 | 成长投入的供给与节奏 | 共享仓库、聚落、任务、世界态 | 把玩家投入转成上述四条支柱的具体机会 |

五条支柱的关系：

- 技能构筑是五条中最核心的一条，因为它最直接决定“单位会做什么”。
- 其它四条不被视为“平行”或“旁支”，而是正式支柱；但它们必须能说明自己与技能构筑的交互路径，否则就是脱节。
- 任何一条支柱的正式状态变化都进入 `CharacterManagementModule` 的写入链，不允许绕过。

### 2.3 运行核心系统

五条成长支柱依赖下面的运行核心系统才能稳定落地：

| 系统 | 定位 | 主要职责 |
| --- | --- | --- |
| `GameSession` | 状态核心 | save/load、内容缓存、全局注册表 |
| `GameRuntimeFacade` | 编排核心 | world / battle / modal / reward 总编排 |
| `BattleRuntimeModule` | 执行核心 | 命令、规则、命中、伤害、状态、时间轴 |
| `CharacterManagementModule` | 成长写入核心 | 把五条支柱的变化正式写回角色真源 |
| `Enemy / Encounter Systems` | 对抗与内容核心 | 构造对抗，把五条支柱的压力面投放给玩家 |

要点：成长五支柱是玩法本体，运行五核心是玩法本体的执行、持久、编排、验证层，两者不是竞争关系。

## 3. 最高设计原则

### 3.1 成长优先于一次性刺激

- 玩家对世界的每一次显著投入（战斗、探索、聚落、任务、研究）最终都应以五条支柱中的正式状态变化结算。
- 纯即时消费（一次性药水爆发、临时 buff）可以存在，但不应替代支柱结构成为循环的主体。

### 3.2 技能是构筑语义核心

- 在五条支柱中，技能支柱承担“构筑语义”的核心角色：
  - 单位会做什么 = 技能
  - 单位怎么构筑 = 技能组合
  - 单位此刻能做什么 = 当前激活技能在战场上的投影
- 其它四条支柱可以有独立的正式语义（属性值、职业身份、装备修正、仓库资源），但都必须能说明自己对技能构筑的影响路径。
- 如果一条支柱长期完全不与技能构筑产生交互，它就开始脱离本项目的玩法核心。

### 3.3 身份包装技能，不替代技能

- 职业、敌人模板、流派标签是对技能组合的组织与承接，不是第一玩法输入。
- 单位的运行时行为应优先参考“实际技能集合”，其次才是身份 / 模板。
- 模板失配时，允许基于技能类型、目标模式、效果类型做通用行为 fallback；这是正式设计组成部分，不是补丁。

### 3.4 支柱之间必须显式交互

- 装备对属性的修正：装备支柱 → 属性支柱（合法基线）。
- 装备对技能的门槛 / 授予 / 标签修饰：装备支柱 → 技能支柱（**必须存在至少一条这样的通路**，不能全部延后）。
- 职业门槛：优先吃技能或属性现实，而不是仅凭身份 ID 判定。
- 任务奖励：必须命中某一条支柱；整体奖励结构中，技能构筑与装备构筑相关收益应占主导。

不允许的情况：

- 一条支柱长期只对自己内部写入、从不与其它支柱交互。
- 一条支柱被绕开：另造“平行能力表”直接进入战斗结算。

### 3.5 战斗是成长支柱的执行场

- 战斗负责位置、命中、伤害、状态、时间轴、资源结算。
- 战斗消费的是五条支柱的当前投影，不是另一套独立能力语言。
- 装备的“锋锐 / 坚韧”、技能的“范围 / 冷却”、职业的“被动”都应视为五条支柱在战斗里的显式接口，而不是战斗系统自造的私有语言。

### 3.6 奖励与投放回流到支柱

- 掉落、任务奖励、聚落服务、研究、训练的正式收益必须回流到至少一条成长支柱。
- 允许存在辅助性收益（金钱、声望、消耗品）——它们应支撑支柱成长，而不是长期替代支柱成长。
- 一个奖励系统若长期产出的东西不能命中任何支柱，它就与主循环断裂。

## 4. 整游戏主循环

本项目的正式主循环，应该理解为下面这条链：

```text
内容投放
-> 获得/抽取技能机会
-> 形成角色或敌人的实际技能集合
-> 形成构筑与能力结构
-> 投影到世界态与战斗态
-> 在战斗/聚落/任务中消耗与验证这套能力
-> 产出奖励、成长、世界反馈
-> 再回流到新的技能状态变化
```

把它拆开看：

1. 内容系统提供技能机会  
世界生成、敌人模板、技能书、任务奖励、研究、聚落服务，决定玩家和敌人会接触到哪些技能。

2. 成长系统把机会沉淀为正式状态  
学习、熟练、升级、融合、核心化、职业绑定，把技能机会变成持久成长。

3. 构筑系统把正式状态组织成当前玩法表达  
激活哪些技能、哪些技能属于核心、职业如何承接、装备如何放大，形成当前 build。

4. 运行时系统把 build 投影成可执行能力  
进入战斗后，角色和敌人都不再以抽象概念存在，而是以当前能力集合存在。

5. 执行系统检验与兑现技能语义  
战斗、任务、据点服务、世界事件，都是这套能力的验证场。

6. 反馈系统再把结果写回技能循环  
奖励、掉落、成长、推进、失败成本，最终都应影响下一轮技能状态。

## 5. 各子系统设计原则

### 5.1 技能内容系统

技能内容系统应负责定义：

- 技能唯一标识与展示信息
- 技能标签
- 战斗配置档
- 属性修正
- 学习条件
- 熟练成长曲线
- 融合/升级关系

这里的设计原则是：

- 技能定义必须是第一真相源。
- 不应并行存在另一套“真正决定玩法但不叫技能”的内容系统。
- 如果一个玩法能力无法映射到技能语义，它要么属于底层战斗规则，要么说明建模错位。

### 5.2 角色成长系统

学习技能在当前游戏系统中的目的，不是单纯让角色多一个按钮，而是把角色成长正式写入统一状态。

因此成长系统必须围绕技能完成：

- 学会
- 熟练
- 升级
- 核心化
- 融合
- 职业承接
- 与技能相关的属性与资源成长

成长系统的原则是：

- 技能是成长主引擎。
- 职业是构筑放大器。
- 属性成长应尽量解释为技能成长与构筑演化的结果，而不是独立漂浮的平行面板系统。

### 5.3 职业系统

职业系统的职责不是定义“这个角色本质上是谁”，而是：

- 把一组技能组织成可识别的成长路径
- 为某类技能提供承接与放大
- 提供职业身份、rank、激活条件、授予收益

职业系统必须服从两个约束：

- 不能与角色实际技能集合冲突。
- 不能取代技能成为角色玩法主语。

也就是说：

- 先有技能现实。
- 再有职业解释。

### 5.4 战斗系统

战斗系统是技能语义的正式执行场。

它的职责是提供：

- 棋盘与空间关系
- 目标合法性
- 资源消耗
- 命中/伤害/状态/地形结算
- 时间轴推进
- 命令预览与执行

战斗系统不应承担的职责是：

- 重新定义角色会什么
- 让单位在没有正式技能依据的情况下获得临时玩法身份
- 在技能系统之外再建一套“更真实的能力语义”

战斗系统的正确位置是：

- 它不是玩法本体。
- 它是玩法本体的执行器。

### 5.5 敌人系统与敌人 AI

敌人系统是这份总纲下非常重要的子系统，但它不是总纲本身。

敌人系统应遵循下面这条原则链：

```text
先技能
-> 后能力
-> 先能力
-> 后模板
-> 模板失配
-> 再回退到技能类型驱动的通用行为
```

具体解释：

- 敌人模板的职责，是提供先验边界与高质量封装：
  - 技能池标签
  - 抽取倾向
  - 常见能力轮廓
  - 推荐行为模板
- 进入战斗后，真正参与决策的第一输入，不是敌人的抽象身份，而是它这次**实际抽到了哪些技能**。
- AI 应先从实际技能集推导当前能力结构，再去匹配模板。
- 若没有合适模板，就直接根据技能类型、目标模式、效果类型、距离需求拼出通用行为。

因此敌人 AI 的系统定位应是：

- 模板不是第一真相源。
- 技能集才是第一真相源。
- 模板只是高质量行为封装。
- fallback 不是补丁，而是纯权重技能生成下的正式组成部分。

### 5.6 世界生成、遭遇与内容投放系统

世界系统、遭遇系统、聚落系统、任务系统，并不直接决定玩法本体是什么；它们决定的是：

- 技能如何被玩家遇到
- 技能如何被敌人持有
- 技能成长如何被节奏化投放
- 哪些 counterplay 被推到玩家面前

它们的核心职责，是构建技能循环的供给面与压力面。

因此这些系统在设计时必须回答：

- 它在投放什么技能机会。
- 它在迫使玩家面对什么技能压力。
- 它是否推动新的构筑选择。

### 5.7 物品、经济与装备系统

经济系统不应成为脱离技能循环的独立 power loop。

更合理的定位是：

- 物品负责技能获取与构筑支持
- 材料负责融合、制作、成长支撑
- 装备负责放大或限制技能构筑
- 商店与制造负责调配技能相关资源

这意味着：

- 技能书是直接的技能入口。
- 装备的意义应尽量与技能标签、职业承接、资源链、战斗表现相连。
- 经济系统应服务于“培养什么技能、维持什么 build、如何应对什么对抗”，而不是只制造脱离技能的数值膨胀。

### 5.8 任务、成就、研究与奖励系统

任务、成就、研究的价值，不在于独立成立一套平行成长体系，而在于把玩家引导回技能循环。

因此它们最适合产出的收益类型是：

- 技能解锁
- 技能熟练度
- 知识门槛
- 职业解锁条件
- 核心构筑机会
- 与技能相关的属性/资源成长

不适合长期占主导的收益类型是：

- 与技能无关的大量永久平面数值
- 无法映射到角色能力结构的抽象奖励

### 5.9 存档、会话与快照系统

从整游戏角度看，持久化系统的关键任务不是“把一切都存下来”，而是稳定保存技能相关正式状态，并允许运行时从它们重建玩法能力。

所以 save/load 的重心应该是：

- 技能进度
- 职业进度
- 核心技能关系
- 构筑相关状态
- 与技能循环相关的资源与奖励队列

而战斗快照、世界快照、headless 快照的设计原则应是：

- 优先暴露技能相关能力事实
- 不优先暴露与技能主循环无关的临时拼装信息

### 5.10 UI 与信息表达系统

UI 不只是展示层，它决定玩家如何理解这个游戏。

如果这份总纲成立，那么 UI 的第一表达维度应该是：

- 我学会了什么技能
- 这些技能成长到什么程度
- 它们构成了什么能力结构
- 这套构筑在战斗里会做什么
- 下一步我还能把哪条技能线往前推进

也就是说，UI 的组织应该优先让玩家按技能和能力理解角色，而不是只按抽象职业标签理解角色。

## 6. 系统红线

后续任何子系统设计，只要碰到下面这些情况，都应判定为偏离总纲。

### 6.1 不允许建立绕过五支柱的平行能力核心

- 任何对玩法主体验产生正式影响的系统，其输入输出都必须命中至少一条成长支柱，并经过该支柱的 owner。
- 另造一套“正式决定玩法但不属于任何支柱”的能力表，就是在五支柱之外分裂主干。

### 6.2 身份与外挂不得替代技能构筑现实

- 职业 ID、敌人模板、流派标签、稀有度标签，不得先于实际技能集 / 属性 / 装备投入运行时决策。
- 装备、buff、地形等外挂能力可以**影响**构筑表达，但不得在战斗决策或构筑展示中**替代**“这个单位实际会什么”作为第一输入。
- 典型违规：让角色“穿上重甲就自动变成前排坦克”、让敌人“因为是 Orc 模板就总是冲锋”。

### 6.3 不允许支柱孤岛

- 一条支柱长期只对自己内部写入、不消费其它支柱的状态、也不向其它支柱输出，就与主循环断裂。
- 典型违规：
  - 装备支柱长期只做“静态属性修正”，从不涉及技能门槛、技能授予、技能标签。
  - 职业支柱长期只做“身份徽章”，从不影响技能或属性构筑。
  - 资源支柱长期只产平面经济，从不命中技能 / 装备 / 职业任何一条。

### 6.4 不允许绕过成长写入核心

- 五条支柱的正式状态变化必须经过 `CharacterManagementModule`。
- 不允许 UI 直接改写 `EquipmentState` / `PartyMemberState`；不允许战斗结算直接向永久成长字段写入。
- 战斗中的临时状态可以由 `BattleRuntimeModule` 独立推进，但结算回写必须走正式通道。

### 6.5 不允许战斗独自定义正式能力语言

- 战斗规则（命中、伤害、状态、时间轴、位置、资源）是正式执行规则，不是正式能力语言。
- 若战斗系统需要表达“这个单位此刻能做 X”，X 必须归到五条支柱的某一条（技能效果、装备授予、职业被动、属性阈值触发）。
- 战斗不能在五支柱之外保留自己的“玩法能力清单”。

### 6.6 不允许奖励与投放长期脱离支柱

- 允许辅助性收益（金钱、声望、消耗品），但奖励系统整体上长期不产生任何支柱变化，就等同于没接入主循环。
- 技能构筑与装备构筑相关收益在奖励结构中的主导地位是**结构性要求**，不是滚动运营建议。

## 7. 新子系统接入检查表

后续任何新子系统，都建议先过这张表。

| 检查问题 | 正确方向 | 预警信号 |
| --- | --- | --- |
| 这个系统命中哪一条成长支柱 | 明确命中 ≥ 1 条支柱，并能说明自己是 owner 还是消费方 | 无法清楚指出命中支柱；或看似命中但不经过支柱 owner |
| 它与技能构筑的交互通路是什么 | 有清楚的门槛 / 授予 / 标签 / 构筑修饰路径 | 长期完全不与技能支柱挂钩 |
| 它消费什么输入 | 实际技能集、属性快照、装备状态、职业记录 | 只消费身份标签、模板名、流派口号 |
| 它改变什么正式状态 | 经过 `CharacterManagementModule` 或对应支柱 owner 写入 | UI 或战斗直接写入永久字段 |
| 它如何进入战斗运行时 | 通过支柱既有接口投影到战斗单位快照 | 在战斗里新造一套“平行能力表” |
| 它的奖励如何回流 | 命中支柱，且技能构筑 / 装备构筑收益占主导 | 长期只产平面资源，与所有支柱断裂 |
| 它如何被 UI 表达 | 让玩家看到支柱状态变化与构筑演化 | 玩家只能看到身份徽章，看不到能力变化 |

表的用法：

- 任何新子系统在立项阶段都应过一遍这张表。
- 一条预警不一定致命，但两条以上预警意味着这个系统还没有真正接入主干。
- 第 2 行“与技能构筑的交互通路”是**强约束**——这一列一旦长期没有答案，就触发 6.3 的“支柱孤岛”红线。

## 8. 这份总纲对后续设计的直接要求

### 8.1 对成长文档

- 必须继续坚持“技能是最小正式成长单位”。
- 职业、属性、核心技能、构筑，都应被解释为技能成长体系的展开。

### 8.2 对战斗文档

- 必须把战斗视为技能语义执行层，而不是独立主玩法。
- 新规则应尽量围绕技能执行、技能互动、技能 counterplay 展开。

### 8.3 对敌人 AI 文档

- 必须默认“实际技能集”是第一输入。
- 敌人模板、行为树、Action 系统都只是对技能能力结构的组织与封装。

### 8.4 对任务、聚落、经济文档

- 必须说明它们如何投放技能机会、推动技能成长、改变构筑选择。
- 如果一个设计无法回答这点，它就还没有接上主循环。

## 9. 结论

这份总纲最终要固定的，不是一句口号，而是一条设计秩序：

- 技能系统是当前项目的玩法核心。
- `GameSession` 是状态核心。
- `GameRuntimeFacade` 是编排核心。
- `BattleRuntimeModule` 是执行核心。
- `CharacterManagementModule` 是成长写入核心。
- 敌人、任务、经济、世界、奖励、UI 都要围绕技能系统组织自己的职责。

因此，今后整个项目的系统设计判断标准应统一成一句话：

> 一个系统如果不能清楚说明自己如何生成、组织、消费、执行或回流技能状态，它就还没有真正接上当前游戏的主干设计。
