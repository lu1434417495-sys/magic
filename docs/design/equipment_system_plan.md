# 角色装备系统设计

更新日期：`2026-04-13`

## 目标

- 把当前已经落地的“共享仓库 + 固定槽位 + 静态属性加成”整理成正式基线。
- 为后续职业限制、双手武器、装备窗口、掉落来源、装备技能与品质系统预留一致的数据与运行时边界。
- 保持装备系统与当前项目主干一致：
  - 真相源在 `PartyState -> PartyMemberState -> EquipmentState`
  - 物品定义仍以 `ItemDef` 为入口
  - 角色属性仍通过 `CharacterManagementModule -> AttributeService` 统一结算
  - 装备的取得与回收仍通过共享仓库完成

## 版本定位

- 当前已实现并可视为正式基线的能力：
  - 固定装备槽位
  - 共享仓库出入仓换装
  - 装备静态属性修正
  - 存档 round-trip
  - headless / 文本命令换装
  - 种子装备与自动化回归
- 本文档在此基础上继续定义“正式装备系统”的下一阶段边界，但不要求一次性全部实现。
- 推荐按阶段推进，而不是为了单一装备规则把仓库、角色成长、战斗快照、UI 一次性重写。
- 本文档明确采用“只面向当前版本”的策略：
  - 不承诺兼容老版本装备存档结构
  - 不承诺兼容旧版 `equipment_state` 形状
  - 如果后续装备 schema 升级，需要显式迁移工具或直接废弃旧存档，而不是在正式逻辑里长期保留兼容分支
- 若后续准备推进装备耐久，请单独参考 [equipment_durability_plan.md](equipment_durability_plan.md)；当前总文档只保留阶段边界，不直接批准实现。

## Summary

- 装备系统以 `PartyState -> PartyMemberState -> EquipmentState` 为唯一真相源。
- 装备物品来源于共享仓库，换装本质上是“仓库与角色槽位之间的受约束转移”。
- 装备状态只保存“槽位 -> item_id / instance_id”，属性、被动、技能授予等效果由物品定义即时展开，不在角色存档中冗余缓存。
- 首版正式系统继续以静态装备为核心；耐久、随机词缀、套装、镶嵌、强化、唯一实例作为后续阶段能力，不混进当前基线。
- 耐久的延后方案与实例化前置要求，见 [equipment_durability_plan.md](equipment_durability_plan.md)。

## 设计原则

- 单一真相源：角色当前装备只允许由 `EquipmentState` 表达，不能在 UI、战斗状态、属性快照里再维护第二份装备表。
- 规则集中：装备合法性由装备规则层和装备服务统一裁决，不允许窗口脚本直接改槽位。
- 与仓库解耦但串联：仓库决定物品库存与容量，装备系统决定角色是否能穿、穿到哪、是否与现有装备冲突。
- 与成长系统并行：装备可以影响属性、标签、临时技能，但不直接改写角色永久成长数据。
- 渐进式扩展：先稳住固定槽位与属性修正，再扩到装备限制、技能授予、品质实例化。

## State Ownership

- `ItemDef`
  - 物品静态定义真相源。
  - 负责声明物品是否属于装备、可进入哪些槽位、有哪些静态修正。
- `EquipmentState`
  - 角色当前装备真相源。
  - 只表达每个槽位当前装了什么，不负责做合法性校验。
- `PartyEquipmentService`
  - 装备规则执行入口。
  - 负责槽位决策、合法性校验、仓库流转、装备摘要输出、属性修正展开。
- `PartyWarehouseService`
  - 仓库真相源与容量规则 owner。
  - 不判断角色职业、体型、装备互斥，只处理库存能否增减。
- `CharacterManagementModule`
  - 消费装备结果，重建角色属性快照、技能表和战斗单位快照。
- `GameRuntimeFacade`
  - 负责命令入口、窗口互斥、战斗中禁换装等运行时约束。

## 作用范围

- 本系统负责：
  - 装备位管理
  - 共享仓库与角色槽位之间的物品流转
  - 装备对角色属性与战斗快照的影响
  - 装备被显式技能效果破坏时的状态更新规则
  - 装备相关 UI 数据结构
  - 装备相关的存档恢复与文本命令入口
- 本系统暂不负责：
  - 消耗品使用
  - 战斗内临时拾取与即时换装
  - 随机掉落表与商店经济平衡
  - 强化、镶嵌、洗练、词缀生成
  - 套装触发与唯一实例追踪

## 槽位模型

- 当前正式槽位：
  - `main_hand`
  - `off_hand`
  - `head`
  - `body`
  - `accessory_1`
  - `accessory_2`
- 首版仍按固定槽位工作，保证：
  - UI 布局稳定
  - 文本快照稳定
  - 存档结构稳定
- 后续扩展不直接删除这些槽位，而是在其上增加“占用关系”语义：
  - 单手武器：通常只占 `main_hand`
  - 盾牌 / 副手法器：占 `off_hand`
  - 双手武器：逻辑上同时占 `main_hand + off_hand`
  - 双饰品仍然是两个独立槽位
- 因此后续推荐区分两个概念：
  - `allowed_slot_ids`：物品允许被点击放入哪些入口槽
  - `occupied_slot_ids`：装备后真实占用哪些槽位
- 当前代码尚未实现 `occupied_slot_ids`，但文档先把它定为正式扩展方向，避免未来继续把“双手武器”硬塞成特殊 if/else。

## 物品模型

- 当前 `ItemDef` 已有并继续沿用的字段：
  - `item_id`
  - `display_name`
  - `description`
  - `icon`
  - `is_stackable`
  - `max_stack`
  - `item_category`
  - `equipment_slot_ids`
  - `attribute_modifiers`
- 装备物品基线约束：
  - `item_category == "equipment"`
  - 至少声明一个合法槽位
  - 首版仍建议不可堆叠，`max_stack == 1`
- 正式系统建议在后续阶段增加但不立即实现的字段：
  - `equipment_type_id`
    - 例如 `sword` / `shield` / `light_armor` / `charm`
  - `equip_requirements`
    - 职业、标签、属性、知识、剧情、体型等复合条件
  - `occupied_slot_ids`
    - 用于双手武器、巨盾、特殊饰品等多槽位占用
  - `exclusive_tags`
    - 用于“盾牌与双手武器互斥”“大型法典与副手法器互斥”
  - `granted_skill_ids`
    - 装备后临时授予技能
  - `granted_passive_tags`
    - 装备后授予抗性、地形亲和、状态免疫等标签
  - `rarity_tier`
    - 品质展示与掉落池权重
  - `break_resistance_percent`
    - 用于装备破坏类技能的额外抗性覆盖值；默认可由 `rarity_tier` 推导
  - `slot_break_weight_override`
    - 允许少数特殊装备覆盖默认部位被选中概率
  - `instance_mode`
    - 标记该物品是共享模板装备还是未来的唯一实例装备

## 装备规则层

### 1. 内容校验

- 装备类物品必须能通过 `ItemContentRegistry` 校验。
- 非装备物品不能进入任何装备流程。
- 槽位声明必须是合法槽位。
- 后续新增的 `occupied_slot_ids`、`exclusive_tags`、`equip_requirements` 也应纳入内容校验，而不是等到运行时报错。

### 2. 角色资格校验

- 当前基线：
  - 不限制职业
  - 不限制体型
  - 不限制属性
  - 不限制知识或成就
- 正式系统下一阶段建议支持以下资格约束：
  - 职业 / 职业标签要求
  - 基础属性阈值要求
  - 知识或成就要求
  - 角色标签要求
  - 体型 / 武器类型适配要求
- 这些限制应在换装确认前统一由 `PartyEquipmentService` 判定，并返回稳定错误码给 UI / headless 命令。

### 3. 槽位决策与互斥

- 当前基线：
  - 若未指定槽位，优先选择第一个空槽
  - 若没有空槽，则覆盖声明顺序中的第一个槽位
- 正式系统下一阶段要把“覆盖单槽”升级为“占用集替换”：
  - 若装备会占用多个槽位，则先找出所有受影响槽位
  - 若被替换装备需要回仓，则必须整体可回滚
  - 同一装备触发多个冲突时，换装操作仍视为一次原子事务
- 典型互斥规则：
  - 双手武器与副手装备互斥
  - 巨盾与双持副手互斥
  - 指定独占标签的饰品之间互斥

### 4. 仓库事务

- 当前基线规则继续成立：
  - 装备时先从共享仓库移除目标物品
  - 若目标槽已有装备，则旧装备尝试回仓
  - 回仓失败时整次换装回滚
  - 卸装前先确认仓库可接收目标物品
- 正式系统继续要求：
  - 多槽位替换仍然必须是原子事务
  - 任一旧装备无法回仓，整次换装失败
  - UI 不得绕开仓库直接写 `EquipmentState`

### 5. 战斗期约束

- 当前运行时已经禁止战斗中换装，这条规则保留。
- 原因：
  - 当前战斗快照在开战时构建
  - 战斗中换装会牵连属性、技能、AI 评估与装备展示同步
- 例外规则：
  - “装备破坏”属于显式战斗技能效果，不视为玩家主动换装
  - 如果某个技能正式定义了可破坏装备，则允许它在战斗结算中直接移除目标槽位装备
- 未来如果要支持战斗内换装，应另开一条正式设计，不在本系统首版范围内偷开后门。

### 6. 装备破坏语义

- 装备破坏是显式技能效果，不等同于耐久损耗，也不等同于普通卸装。
- 首版推荐规则：
  - 单次效果默认最多破坏 1 件装备
  - 只从目标当前已装备槽位中选择候选
  - 破坏成功后，装备直接从 `EquipmentState` 中移除
  - 被破坏装备不回共享仓库
- 概率模型建议拆成两层：
  - 槽位层：不同部位有不同的被抽中权重或破坏系数
  - 装备层：不同 `rarity_tier` 或 `break_resistance_percent` 提供不同抗性
- 默认方向：
  - 高稀有度装备更难被破坏
  - 主手、副手、头部比身躯和饰品更容易成为破坏目标
- 若未来启用唯一实例装备：
  - 破坏成功时应销毁对应实例
  - 不保留为“已损坏但仍占槽”的半状态，除非后续正式引入耐久/修理系统

## 装备效果模型

### 已实现基线

- 静态属性修正来自 `ItemDef.attribute_modifiers`
- 装备修正与职业、技能修正共走 `AttributeService`
- 装备不会直接改写角色基础属性

### 正式扩展方向

- 装备效果分三层处理：
  - 静态层：属性、抗性、命中、闪避、移动、视野等常驻修正
  - 授予层：临时授予主动技能 / 被动技能 / 标签
  - 规则层：改变装备可用动作、伤害类型、攻击距离、状态抗性等
- 其中：
  - 静态层优先继续复用 `AttributeModifier`
  - 授予层建议通过 `granted_skill_ids` 并在角色快照构建时合并
  - 规则层不得散落在 UI 或战斗面板里，应统一落在角色快照或战斗单位生成链上

## 与角色成长系统的关系

- 装备不是永久成长，不进入角色的固定技能学习记录或职业历史。
- 装备授予的主动技能若进入后续实现，应视为“临时可用技能”：
  - 卸下装备后失效
  - 不提升该技能的永久等级
  - 不参与技能合并的永久拥有关系
- 装备导致的属性变化仍应能影响：
  - 当前属性快照
  - 开战时生成的战斗单位属性
  - 命中、伤害、行动顺序等派生结果

## 与战斗系统的关系

- 开战前：
  - `CharacterManagementModule` 基于当前 `EquipmentState` 构建属性快照与技能快照
- 开战时：
  - 战斗单位读取已结算后的属性与可用技能
- 战斗中：
  - 默认不允许换装
  - 战斗日志与详情面板只读展示装备结果，不反向改装备
- 战斗后：
  - 战斗结果默认不直接改写装备
  - 显式技能效果造成的装备破坏是正式例外，应在结算链路中写回装备真源

## 与获取来源的关系

- 当前基线：
  - 装备物品主要通过种子内容和测试命令注入共享仓库
- 正式系统建议后续统一接入四类来源：
  - 战利品掉落
  - 商店购买
  - 任务奖励
  - 制作 / 合成
- 原则：
  - 装备获取先进入共享仓库，再由玩家决定是否换装
  - 不推荐把绝大多数装备做成“自动穿上”
  - 自动装备只适合作为特殊奖励脚本或新手教学流程

## UI 设计

### 当前状态

- `PartyManagementWindow` 只展示装备摘要，不负责换装
- 仓库窗口只展示共享仓库，不负责角色槽位对比

### 正式装备窗口目标

- 推荐新增独立装备管理视图，而不是继续把复杂换装逻辑塞进当前仓库窗口
- 推荐布局：
  - 左侧：队伍成员列表
  - 中间：当前成员的装备槽位、当前属性摘要、职业/标签限制
  - 右侧：共享仓库中过滤后的可装备候选
  - 底部：换装前后对比、错误提示、来源说明
- 必须支持的交互：
  - 选中成员
  - 点击槽位查看当前已装备物品
  - 过滤出该槽位可装备物品
  - 预览属性变化
  - 明确显示被替换物品会回到仓库
  - 明确显示无法装备的原因
- 首版 UI 不要求拖拽，列表点击即可

## 文本命令与自动化入口

- 当前已存在并继续保留：
  - `warehouse add <item_id> <quantity>`
  - `party equip <member_id> <item_id> [slot_id]`
  - `party unequip <member_id> <slot_id>`
- `snapshot party` 继续输出：
  - `attributes`
  - `equipment`
  - `equipment_count`
- 后续若加入装备需求或冲突规则，headless 命令必须同步输出稳定错误信息，确保回归脚本能断言失败原因。

## 持久化策略

- 当前正式存档结构继续保持：
  - `PartyState`
    - `PartyMemberState`
      - `equipment_state`
        - `equipped_slots.<slot_id> = <item_id>`
- 版本策略：
  - 装备系统文档口径明确不支持老版本
  - 只保证当前正式结构的读写与同版本 round-trip
  - 不再要求继续支持历史 `equipment_state` 兼容恢复
- 若未来引入唯一实例装备，推荐升级为：
  - `equipped_slots.<slot_id> = { item_id, instance_id }`
- 若该结构启用，视为新的正式版本；旧版本存档默认不在支持范围内。

## 阶段规划

### Phase 1：稳定基线

- 固定槽位
- 共享仓库联动
- 静态属性修正
- 当前版本存档稳定
- headless / 自动化回归

### Phase 2：正式可玩装备系统

- 图形化装备窗口
- 职业 / 属性 / 标签装备限制
- 双手武器与占用槽位模型
- 装备前后属性对比
- 掉落 / 商店 / 奖励接入共享仓库

### Phase 3：装备深度化

- 装备授予技能
- 品质分级与更丰富的装备类型
- 独占标签与更复杂的互斥规则
- Named gear 或唯一实例装备
- 装备破坏抗性与特殊保护标签

### Phase 4：高复杂度扩展

- 随机词缀
- 套装效果
- 强化、镶嵌、耐久
  - 耐久草案见 [equipment_durability_plan.md](equipment_durability_plan.md)
- 战斗中有限换装或特化职业机制

## 测试计划

### 当前必须持续通过的基线测试

- 装备种子物品通过 `ItemContentRegistry` 校验
- 装备 / 卸装后仓库数量与角色槽位状态守恒
- 多槽位饰品自动装备时优先填空槽
- 装备属性能进入 `AttributeService` 快照，并在卸装后回退
- `PartyState` round-trip 后保留全部装备槽位
- 文本命令链路能够执行加仓、装备、卸装并反映到快照

### Phase 2 新增测试项

- 职业或属性不满足时拒绝装备并返回稳定错误码
- 双手武器会正确占用 `main_hand + off_hand`
- 装备双手武器时，副手已有装备会按原子事务整体替换
- 因仓库容量不足导致回仓失败时，整次换装完整回滚
- 装备窗口的候选过滤、属性预览与最终结算一致

### Phase 3 新增测试项

- 装备授予技能只在装备期间有效
- 卸装后临时技能被正确移除
- 独占标签冲突会稳定阻止非法组合
- 唯一实例装备在存档 round-trip 后保持实例身份
- 装备破坏效果会正确移除目标槽位且不回仓
- 高稀有度装备的破坏概率低于低稀有度装备
